>_SIGNAL-AGNOSTIC PATCH GRAPH

The shared cable-routing core behind the plugin stack. Knows nothing about audio or MIDI. Private core.

Concept

It owns the parts that are identical in every modular plugin and nothing else, so zpwr-fx, zpwr-synth, and zpwr-midi-fx share one routing engine instead of three copies:

  • Routing — nodes wired by source ids, fan-out, feedback.
  • Evaluation — topological order each rebuild; cycles resolve with a one-sample delay.
  • Mod matrix — every node param has a (source, depth) modulation.
  • Per-cable gain + colour.
  • Lock-free live edits (atomic params) + atomic graph swap on structural edits.
  • JSON serialisation of a whole patch, with versioning + source-id migration.
  • ScriptEngine — a small RT-safe expression VM, exposed as the Expr module.

Signal-agnostic by design

The graph is templated on the signal type carried between nodes (a SignalTraits<S> policy): float for audio (zpwr-fx, zpwr-synth) and a note-event stream for MIDI (zpwr-midi-fx). Each host supplies a ModuleRegistry of node types and the external source values per sample; the core resolves block outputs and leaves the rest to the host.

ext sources  →  [ Node 0 ]  →  [ Node 1 ]  →  …  →  outputs
  In L/R         (registry-defined compute, per-param mod matrix)
  noise/keys     S signal out  +  float scalar projection (mod value)
  MIDI/MPE

Shared WebEditor

zpc::WebEditor<Engine> is the WebView backend every host shares — catalog / patch JSON, BinaryData serving, preset I/O, ~30 native functions. Soft knobs are an expandable pool with a fixed automatable ceiling; the UI's +/− controls grow the active count without shifting host source ids. EZ mode exposes an ⚡ EZ WIRE button that connects the standard signal path for linear hosts.

Dependency

C++20juce_core onlyheader + lib

Consumers add it via add_subdirectory and link zpwr::patch_core. No audio, MIDI, GUI, or networking coupling — JUCE's curl path is disabled.
