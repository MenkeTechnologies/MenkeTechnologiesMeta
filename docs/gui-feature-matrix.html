>_EXECUTIVE SUMMARY

Every full Tauri app mounts the same zgui-core shell, so the power-user surfaces — GUI scripting bus, tmux/powerline status bar, tmux tiling WM, editor vim, stryke hooks editor — are near-universal by construction (the identical cyberpunk HUD across apps is intentional). The signal is at the wiring level and the outliers: several surfaces ship in zgui-core but must be init()'d per app, and two apps step outside the webview shell entirely — zterminal (native Rust terminal, native tmux) and the JUCE plugins (no webview at all).

&TRACK B — JUCE PLUGINS

zpwr-daw (engine) · zpwr-synth · zpwr-fx · zpwr-midi-fx. No window.ZGui, no Tauri invoke. The automation bus, powerline bar, and tmux WM are not wired for the JUCE surface — the Track B bus substrate (C++/C-ABI surface, per-plugin-instance socket addressing) is still unbuilt (see GUI_AUTOMATION_BUS_CHECKLIST.md §0B). zpwr-daw's Tauri shell carries monaco-vim + the hooks editor through its embedded ztranslator-core webview; the JUCE ClipEngine half does not.

$GUI SCRIPTING IS LIVE, NOT PLANNED

GUI_SCRIPT_ACTIONS.md is regenerated from each app's verb source by bin/gen-gui-actions.sh and currently lists 3073 engine verbs across 11 apps + 15 shared appShell verbs. Every full Tauri app opens its socket (serve("<app>") + bus::start, verified at the call site). The fwd apps run a webview-forward bus.rs: every verb is forwarded to the webview, which tries a registered ZGui.automation verb then falls back to invoke(verb,args) — so the app's own Tauri commands are already bus-callable by name and surface() advertises them for discovery. Only traderview and zwire ship a typed register({app,verbs}) surface today.

# drive an app by name from a stryke script over the bus
App::open("zpdf")->call("add_text", {page: 1, x: 72, y: 700, text: "REVIEWED"})
App::open("zcite")->get("selection")   # -> the selected reference

# or from a hook running INSIDE the app
App::here()->verbs()                    # this app's verb surface
