Early-stage, stated honestly. zoffice-core reads ODF and OOXML packages today; editing, saving, formula evaluation, presentation authoring, and the Base database backend are typed Error::NotImplemented stubs. The module map below marks exactly what is implemented now versus planned. Status is derived from the actual fn symbols in src/ — a capability is done only once its body exists, and the test suite pins that every unimplemented capability returns a typed error rather than panicking or silently succeeding.

The lib root adds the Engine JSON dispatcher (info, open auto-detect, and the per-app <app>.open / .text / calc.cells commands) plus App / app_for extension routing; ffi exposes the C ABI (zoffice_invoke / zoffice_string_free) in the staticlib/cdylib artifacts.

No LibreOffice/UNO runtime, no C++ FFI, no copyleft. Crate types: rlib for native Rust/Tauri hosts; staticlib + cdylib for the C ABI. Two crates carry the whole reader/writer surface because every office document is a ZIP of XML parts.

Verification

The test suite builds minimal valid office packages in memory (ODF or OOXML — both are just ZIP-of-XML), writes each to a temp file, and drives the public API: docx and odt paragraph extraction, xlsx shared-string + cell-reference resolution, ods cell parsing with computed refs, pptx multi-slide text, odg page/shape counting, the Engine JSON dispatcher plus its info descriptor, and a guard that every unimplemented capability (evaluate, save_xlsx, save_docx, add_slide, base.open, unknown commands) returns a typed error. No external fixtures; runs identically in a headless Linux CI.
